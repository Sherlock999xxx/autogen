## 1. Обзор проекта

**AutoGen** - это мощный фреймворк от Microsoft для создания мультиагентных AI-приложений, которые могут работать автономно или совместно с людьми. Проект построен на архитектуре событийно-управляемой модели акторов (Actor Model) и предоставляет все необходимое для создания сложных систем с несколькими AI-агентами.

### Ключевые возможности:
- Создание AI-агентов с поддержкой различных LLM (OpenAI, Azure, Anthropic, Ollama и др.)
- Оркестрация команд агентов с различными паттернами взаимодействия
- Безопасное выполнение кода в изолированных средах
- Распределённое выполнение с использованием gRPC
- GUI для прототипирования без кода (AutoGen Studio)
- Кросс-платформенная поддержка (Python и .NET)

## 3. Архитектура проекта

### 3.1. Трёхслойная архитектура

AutoGen состоит из трёх основных слоёв:

```
┌─────────────────────────────────────────────────────────┐
│ AgentChat (Высокоуровневый API)                         │
│ - Готовые агенты (Assistant, UserProxy, CodeExecutor)  │
│ - Абстракции команд (GroupChat, Swarm, DiGraph)        │
│ - Типы сообщений для диалогов                          │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ AutoGen Core (Среднеуровневая среда выполнения)        │
│ - Протокол Agent & AgentRuntime                        │
│ - Маршрутизация через Topics & Subscriptions           │
│ - SingleThreadedAgentRuntime                           │
│ - Типы агентов (RoutedAgent, BaseAgent, ClosureAgent) │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ AutoGen Ext (Расширения и интеграции)                  │
│ - Распределённая среда выполнения (gRPC)               │
│ - Клиенты LLM (OpenAI, Azure, Anthropic, Ollama)      │
│ - Исполнители кода (Docker, Local, Jupyter, Azure)    │
│ - Реализации памяти и инструментов                     │
└─────────────────────────────────────────────────────────┘
```

### 3.2. Архитектурная модель: Actor Model

AutoGen построен на модели акторов, где:
- **Агенты** - независимые, параллельные сущности, обрабатывающие сообщения асинхронно
- **Коммуникация** происходит исключительно через передачу сообщений
- **Нет разделяемого состояния** между агентами
- Обеспечивает **распределённость, масштабируемость и отказоустойчивость**

### 3.3. Основные компоненты

#### 3.3.1. AutoGen Core (`autogen-core`)

**Ключевые интерфейсы:**

- **Agent (Protocol)** - базовый протокол агента
  - `metadata`: метаданные агента
  - `id`: уникальный идентификатор
  - `on_message()`: обработчик сообщений
  - `save_state()` / `load_state()`: персистентность

- **AgentRuntime (Protocol)** - среда выполнения агентов
  - `send_message()`: прямая отправка сообщения агенту (RPC)
  - `publish_message()`: широковещательная отправка (pub-sub)
  - `register_factory()`: регистрация фабрики агентов
  - `add_subscription()`: управление маршрутизацией

**Реализации агентов:**

1. **BaseAgent** - абстрактный базовый класс
2. **RoutedAgent** - агент с декораторами для маршрутизации:
   - `@message_handler`: обработка RPC и событий
   - `@event`: обработчик событий (без ответа)
   - `@rpc`: RPC обработчик (с ответом)
3. **ClosureAgent** - легковесный агент на основе функции
4. **ToolAgent** - агент для выполнения инструментов

**Маршрутизация сообщений:**

- **TopicId** - идентификатор темы (`type/source`)
- **Subscription** - правила сопоставления тем:
  - `TypeSubscription`: точное совпадение типа
  - `TypePrefixSubscription`: префикс типа
  - `DefaultSubscription`: все сообщения

**Сериализация:**

- `PydanticJsonMessageSerializer` - для Pydantic моделей
- `DataclassJsonMessageSerializer` - для dataclass
- `ProtobufMessageSerializer` - для Protobuf
- `SerializationRegistry` - центральный реестр

**Среда выполнения:**

- **SingleThreadedAgentRuntime** - основная реализация
  - Обрабатывает сообщения из асинхронной очереди последовательно
  - Конверты сообщений: SendMessageEnvelope, PublishMessageEnvelope
  - RunContext для управления жизненным циклом

#### 3.3.2. AutoGen AgentChat (`autogen-agentchat`)

**Типы сообщений:**

- **BaseChatMessage** - базовый класс сообщений
  - `TextMessage` - текстовое сообщение
  - `MultiModalMessage` - изображения + текст
  - `ToolCallMessage` - вызов инструмента
  - `ToolCallResultMessage` - результат инструмента
  - `StopMessage` - сигнал завершения

**Интерфейс ChatAgent:**

- **ChatAgent (Protocol)** - протокол разговорного агента
  - `name`: уникальное имя
  - `description`: описание возможностей
  - `on_messages()`: обработка новых сообщений
  - `on_reset()`: сброс состояния
  - `on_messages_stream()`: потоковый вывод

**Готовые агенты:**

1. **AssistantAgent** (79KB реализация)
   - Агент на основе LLM
   - Поддержка вызова функций/инструментов
   - Системные промпты и контроль температуры
   - Потоковый вывод

2. **UserProxyAgent**
   - Симулирует участие человека
   - Подтверждение выполнения инструментов
   - Сбор ручного ввода

3. **CodeExecutorAgent** (40KB)
   - Выполнение блоков кода
   - Обработка ошибок и парсинг вывода

4. **MessageFilterAgent**
   - Фильтрация/трансформация сообщений
   - Логика маршрутизации

5. **SocietyOfMindAgent**
   - Множественные внутренние агенты
   - Возможность саморефлексии

**Абстракции команд (Teams):**

1. **BaseGroupChat** - базовый класс для команд
   - Управление участниками
   - Условия завершения
   - Ограничение по количеству ходов

2. **RoundRobinGroupChat**
   - Агенты говорят по очереди
   - Простой и предсказуемый

3. **SelectorGroupChat**
   - LLM выбирает следующего говорящего
   - Динамический выбор на основе контекста

4. **Swarm**
   - Маршрутизация на основе handoff
   - Один агент передаёт управление другому

5. **MagenticOneGroupChat**
   - Оркестрированная многоэтапная система
   - Специализирована для сложных задач

6. **DiGraphGroupChat** (на основе графа)
   - Направленный ациклический граф (DAG) агентов
   - Определение сложных workflow с помощью DiGraphBuilder

**Условия завершения:**

- `MaxTurnsTermination` - ограничение по количеству ходов
- `StopMessageTermination` - остановка при StopMessage
- `TextMentionTermination` - остановка при упоминании ключевого слова
- `AndTermination` / `OrTermination` - комбинирование условий

#### 3.3.3. AutoGen Ext (`autogen-ext`)

**Клиенты моделей (`models/`):**

- **OpenAI**: `OpenAIChatCompletionClient`, `AzureOpenAIChatCompletionClient`
- **Anthropic**: `AnthropicChatCompletionClient`
- **Ollama**: `OllamaChatCompletionClient`
- **Google**: Gemini клиенты
- **Azure**: Azure AI клиенты

Все реализуют `ChatCompletionClient`:
- `create()`: асинхронный вызов LLM
- Поддержка потокового вывода
- Подсчёт токенов (`count_tokens()`)

**Исполнители кода (`code_executors/`):**

1. **DockerCommandLineCodeExecutor** (по умолчанию, наиболее безопасный)
   - Выполнение в изолированном Docker контейнере
   - Нет доступа к системе
   - Автоматическая очистка

2. **LocalCommandLineCodeExecutor**
   - Локальное выполнение
   - Быстрее, но менее безопасно
   - Полный доступ к системе

3. **JupyterNotebookCodeExecutor**
   - Использует Jupyter kernel
   - Состояние сохраняется между вызовами
   - Подходит для исследовательского анализа

4. **AzureContainerCodeExecutor**
   - Использует Azure Container Instances
   - Облачная изоляция
   - Удалённое выполнение

**Распределённая среда выполнения (`runtimes/grpc/`):**

- **WorkerRuntime** - распределённый агент на основе gRPC
- **WorkerRuntimeHost** - хост для распределённых агентов
- Protocol Buffers для сериализации
- Позволяет агентам работать на разных машинах

**Дополнительные компоненты:**

- Модули аутентификации
- Реализации памяти (ChromaDB, Mem0, Redis)
- Инструменты и tool агенты
- Бэкенды кэш-хранилищ
- Утилиты UI

## 4. Как это работает

### 4.1. Жизненный цикл агента

```
1. Создание: Определение класса/фабрики агента
2. Регистрация: Runtime.register_factory() или register_agent_instance()
3. Подписка: Runtime.add_subscription() - определение обрабатываемых сообщений
4. Привязка: runtime.bind_id_and_runtime(agent_id, runtime)
5. Обработка: on_message() вызывается средой выполнения
6. Состояние: save_state() / load_state()
7. Очистка: close() вызывается при завершении среды
```

### 4.2. Поток маршрутизации сообщений

```
Агент A отправляет сообщение:
  → Агент A вызывает runtime.send_message(msg, recipient_id)
    ИЛИ runtime.publish_message(msg, topic_id)

Runtime (SingleThreadedAgentRuntime):
  → Получает сообщение
  → Если send_message(): Создаёт SendMessageEnvelope, добавляет в очередь
  → Если publish_message():
    - Проверяет все подписки на совпадение с topic_id
    - Для каждой совпадающей подписки:
      - Сопоставляет тему с агентом через subscription.map_to_agent()
      - Создаёт отдельное сообщение для каждого агента
    - Добавляет PublishMessageEnvelopes в очередь

RunContext обрабатывает очередь:
  → Извлекает конверт из очереди
  → Находит целевой агент(ы)
  → Вызывает agent.on_message(message, context)
  → Ожидает ответа
  → Для RPC: Разрешает future с ответом
  → Для publish: Продолжает обработку очереди
```

### 4.3. Пример работы команды

```
Team (например, RoundRobinGroupChat):
  ↓
  Управляет участниками (агентами)
  ↓
  Оборачивает каждого ChatAgent в ChatAgentContainer
  ↓
  Создаёт внутренний SequentialRoutedAgent как менеджера
  ↓
  Менеджер оркеструет обмен репликами:
    1. Отправляет сообщение Агенту A
    2. Агент A обрабатывает, возвращает Response
    3. Извлекает chat_message из Response
    4. Публикует всем остальным агентам (через тему)
    5. Каждый агент получает через on_messages()
    6. Следующий ход с Агентом B
    7. Повтор до выполнения условия завершения
```

### 4.4. Выполнение кода

```
1. Агент генерирует код
2. Извлечение CodeBlock(code: str, language: str)
3. Передача в CodeExecutor.execute_code_blocks()
4. CodeExecutor безопасно выполняет код
5. Возвращает CodeResult(exit_code: int, output: str)
6. Агент обрабатывает вывод, может сгенерировать следующее сообщение
```

## 6. Использование

### 6.1. Простейший пример (Hello World)

```python
import asyncio
from autogen_agentchat.agents import AssistantAgent
from autogen_ext.models.openai import OpenAIChatCompletionClient

async def main() -> None:
    model_client = OpenAIChatCompletionClient(model="gpt-4o")
    agent = AssistantAgent("assistant", model_client=model_client)
    print(await agent.run(task="Say 'Hello World!'"))
    await model_client.close()

asyncio.run(main())
```

### 6.2. Двухагентный диалог

```python
import asyncio
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_ext.models.openai import OpenAIChatCompletionClient

async def main() -> None:
    model_client = OpenAIChatCompletionClient(model="gpt-4o")

    agent1 = AssistantAgent(
        "agent1",
        model_client=model_client,
        system_message="You are a helpful assistant."
    )

    agent2 = AssistantAgent(
        "agent2",
        model_client=model_client,
        system_message="You are a critic who provides constructive feedback."
    )

    team = RoundRobinGroupChat([agent1, agent2], max_turns=5)
    result = await team.run(task="Discuss the benefits of AI")
    print(result)

asyncio.run(main())
```

### 6.3. Агент с выполнением кода

```python
import asyncio
from autogen_agentchat.agents import AssistantAgent, CodeExecutorAgent
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_ext.models.openai import OpenAIChatCompletionClient
from autogen_ext.code_executors.docker import DockerCommandLineCodeExecutor

async def main() -> None:
    model_client = OpenAIChatCompletionClient(model="gpt-4o")

    # Создание агента-программиста
    coder = AssistantAgent(
        "coder",
        model_client=model_client,
        system_message="You write Python code to solve problems."
    )

    # Создание агента-исполнителя кода
    executor = CodeExecutorAgent(
        "executor",
        code_executor=DockerCommandLineCodeExecutor()
    )

    team = RoundRobinGroupChat([coder, executor], max_turns=10)
    result = await team.run(task="Calculate the first 10 Fibonacci numbers")
    print(result)

asyncio.run(main())
```

```python
        system_message="You are a math expert.",
        description="A math expert assistant.",
        model_client_stream=True,
    )
    math_agent_tool = AgentTool(math_agent, return_value_as_last_message=True)

    # Эксперт по химии
    chemistry_agent = AssistantAgent(
        "chemistry_expert",
        model_client=model_client,
        system_message="You are a chemistry expert.",
        description="A chemistry expert assistant.",
        model_client_stream=True,
    )
    chemistry_agent_tool = AgentTool(chemistry_agent, return_value_as_last_message=True)

    # Главный агент с доступом к экспертам
    agent = AssistantAgent(
        "assistant",
        system_message="You are a general assistant. Use expert tools when needed.",
        model_client=model_client,
        model_client_stream=True,
        tools=[math_agent_tool, chemistry_agent_tool],
        max_tool_iterations=10,
    )

    result = await agent.run(task="What is the integral of x^2?")
    print(result)

asyncio.run(main())
```

## 10. Лучшие практики

### 10.1. Безопасность

1. **Всегда используйте Docker для выполнения кода**
   - Изоляция от основной системы
   - Ограничение ресурсов
   - Автоматическая очистка

2. **Управление секретами**
   - Используйте переменные окружения
   - Не храните API ключи в коде
   - Используйте .env файлы (с .gitignore)

3. **Валидация ввода**
   - Используйте Pydantic для валидации
   - Проверяйте пользовательский ввод
   - Ограничивайте размер ввода

### 10.2. Производительность

1. **Кэширование**
   - Используйте Redis или DiskCache
   - Кэшируйте ответы LLM
   - Кэшируйте результаты инструментов

2. **Параллелизм**
   - Используйте async/await
   - Параллельные вызовы LLM где возможно
   - Распределённое выполнение для масштабирования

3. **Оптимизация токенов**
   - Используйте подходящие модели (не всегда нужна GPT-4)
   - Ограничивайте контекст
   - Используйте сжатие промптов

### 10.3. Надёжность

1. **Обработка ошибок**
   - Используйте try/except блоки
   - Логируйте ошибки
   - Реализуйте retry логику

2. **Тайм-ауты**
   - Устанавливайте разумные тайм-ауты
   - Используйте CancellationToken
   - Обрабатывайте зависшие операции

3. **Состояние и персистентность**
   - Используйте save_state/load_state
   - Сохраняйте чекпоинты
   - Реализуйте восстановление после сбоев

### 10.4. Мониторинг

1. **Логирование**
   - Структурированное логирование
   - Разные уровни (DEBUG, INFO, ERROR)
   - Централизованный сбор логов

2. **Трассировка**
   - OpenTelemetry интеграция
   - Jaeger для визуализации
   - Отслеживание производительности

3. **Метрики**
   - Использование токенов
   - Время выполнения
   - Количество ошибок

## 13. FAQ (Часто задаваемые вопросы)

### Q: Какую модель использовать?
**A:** Зависит от задачи:
- GPT-4o - для сложных задач
- GPT-4o-mini - для простых задач (дешевле)
- Claude - для длинных контекстов
- Llama/Mistral через Ollama - для локального использования

### Q: Безопасно ли выполнение кода?
**A:** Да, при использовании DockerCommandLineCodeExecutor. Код выполняется в изолированном контейнере без доступа к основной системе.

### Q: Можно ли использовать без OpenAI?
**A:** Да, поддерживаются Azure, Anthropic, Ollama, Gemini и другие провайдеры.

### Q: Как ограничить количество токенов?
**A:** Установите max_tokens при создании модели и max_turns для команд.

### Q: Поддерживается ли streaming?
**A:** Да, используйте model_client_stream=True и on_messages_stream().

### Q: Как добавить свой агент?
**A:** Наследуйтесь от BaseChatAgent и реализуйте on_messages().

### Q: Можно ли использовать в production?
**A:** Да, но учитывайте:
- Используйте мониторинг и логирование
- Реализуйте обработку ошибок
- Используйте кэширование
- Настройте ограничения ресурсов

### Q: Стоимость использования?
**A:** Зависит от выбранной модели и объёма использования. Используйте models_usage для отслеживания.
